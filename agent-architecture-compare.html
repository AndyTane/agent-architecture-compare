Nanobot / NanoClaw / OpenClaw 架构对比

基于本地仓库与当前 OpenClaw 实际运行链路整理，重点展示三者的真实执行路径、控制面差异与适用场景。

Nanobot = 蓝色系
NanoClaw = 紫色系
OpenClaw = 绿色系

一屏结论

Nanobot

定位是“超轻量但有完整 agent 能力”的平台。不是 demo，而是已经有 channels、session、cron、heartbeat、gateway、MCP 的实战框架。

  • 适合研究、教学、快速二开
  • 强项：轻量 + 迭代快 + Python 生态
  • 短板：宿主机控制、统一控制面深度弱于 OpenClaw

NanoClaw

更像 TypeScript 版轻量骨架。模块层次很清楚，适合拿来学习 agent platform 怎么拆层。

  • 适合快速读结构、自己重写、做精简版系统
  • 强项：整洁、清晰、容易上手
  • 短板：工具面和平台成熟度目前最薄

OpenClaw

不只是 agent loop，而是完整的 Gateway / Tool / Node / Browser / Canvas / Messaging 平台。

  • 适合生产级个人代理、跨渠道、跨设备控制
  • 强项：统一控制面、外设能力、消息面最强
  • 短板：复杂度最高，认知负担也最高

对比总表

维度 Nanobot NanoClaw OpenClaw
语言 Python TypeScript / Node.js Node.js 平台体系
主定位 轻量但完整的个人 AI assistant 超轻量骨架化 agent 平台 多渠道、多工具、多节点的统一代理平台
核心切入点 agent-first agent-first gateway-first + agent runtime
工具层 已有 MCP 与多能力 基础工具为主 最全，含 browser / canvas / node / messaging
渠道能力 较完整且增长快 已有 Telegram / Discord / DingTalk 等 最成熟、统一到 Gateway
适合场景 研究、轻量生产、快速迭代 学习架构、定制极简平台 真实生产、跨设备、宿主机与浏览器控制

最终判断

NanoClaw = 最容易看懂的骨架；
Nanobot = 轻量但已经有完整闭环；
OpenClaw = 真正的平台级系统，核心差异不是模型，而是 Gateway + Tools + Nodes + Browser + Messaging 这一整层控制面。
